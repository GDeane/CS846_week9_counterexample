## General Overview

The `UserSessionManager` class is a comprehensive, monolithic utility for user authentication, session management,
password handling, email notifications, and some basic HTML rendering. The code is well-structured, with clear
separation of concerns via method grouping and extensive inline documentation. The use of constants for configuration
and the inclusion of security best practices (e.g., password hashing, rate limiting, audit logging) are commendable.

---

## Strengths

1. **Security Best Practices**

- Uses PBKDF2-HMAC-SHA256 for password hashing with a strong salt and high iteration count.
- Implements rate limiting and account lockout to mitigate brute-force attacks.
- Sanitizes user input before rendering in HTML to prevent XSS.
- Uses parameterized SQL queries to prevent SQL injection.

2. **Code Organization**

- Logical grouping of methods (DB, password, validation, CRUD, sessions, email, audit, UI).
- Good use of docstrings and inline comments.
- Constants for configuration make the class easy to adapt.

3. **Robustness**

- Handles database errors with logging and rollback.
- Queues failed emails for retry.
- Provides audit logging for critical actions.

4. **Thread Safety**

- Uses a threading lock for shared in-memory structures.

5. **User Experience**

- HTML rendering methods provide a clean, modern UI.
- Error messages are user-friendly and informative.

---

## Areas for Improvement

1. **Single Responsibility Principle**

- The class is too large and handles too many concerns (DB, email, HTML, business logic). Consider splitting into
  smaller classes/modules:
    - Database access layer
    - Email service
    - Session manager
    - HTML renderer

2. **Testing and Extensibility**

- The tight coupling of logic and side effects (e.g., sending emails directly in user creation) makes unit testing
  harder.
- Consider dependency injection for the email sender and database connection for easier mocking/testing.

3. **Error Handling**

- Some methods return dicts with "success"/"error" keys, others raise exceptions. Standardize error handling (prefer
  exceptions for errors, return values for results).
- Logging is present, but consider more granular log levels and structured logging for audit trails.

4. **Database Connection Management**

- The class maintains a persistent SQLite connection. For web apps, consider using a connection pool or context manager
  to avoid concurrency issues.

5. **HTML Rendering**

- Inline HTML in Python is hard to maintain. Consider using a template engine (e.g., Jinja2) for rendering.
- CSS is embedded in HTML strings; externalize styles for maintainability.

6. **Security**

- SMTP credentials are hardcoded. Move secrets to environment variables or a secure config.
- No CSRF protection in rendered forms (if used in a web context).
- No account verification or password reset flows.

7. **Scalability**

- In-memory caches (`_session_cache`, `_login_attempts`, `_email_queue`) are not persistent and will not scale across
  processes or servers.
- For production, consider Redis or another shared cache.

8. **Type Annotations**

- Good use of type hints, but some return types could be more specific (e.g., `Optional[Dict[str, Any]]` could be a
  custom User type).

- Use `secrets.compare_digest` for all sensitive comparisons.

from datetime import datetime, timedelta

def add_days(date_str: str, days: int) -> str:
    """
    Input:  date_str in 'YYYY-MM-DD'
    Output: date_str in 'YYYY-MM-DD' after adding days
    """
    print(f"[DEBUG] Incoming date_str: {date_str}")

    try:
        dt = datetime.strptime(date_str, "%Y-%m-%d")
    except ValueError as ve:

        result_date = dt + timedelta(days=days)
    except Exception as e:
